Critical = security exploit path or data-loss live in `main`.
High = correctness, security-defense-in-depth, or test-coverage gap on a load-bearing pillar.
Medium = real risk but mitigated, deferred, or behind a flag.
Low = code-smell, drift, or minor robustness gap.
Info = noted for context, no action required.

**Standout positives** (so the report doesn't read as alarmist):

- No `tx.origin`, no `delegatecall(userInput)`, no reentrancy gap, no `block.timestamp` for randomness in `contracts/`.
- No SQL string concatenation; every query goes through Drizzle / `sql\`...\`` placeholders.
- No `verify=False` / `rejectUnauthorized: false` outside `require`-mode dev fallback (no `verify-full` bypass).
- All 43 fraud patterns have both a doc page in `docs/patterns/P-*.md` and code in `packages/patterns/src/`.
- Migrations all have a paired `.down.sql`.
- Currency stored as integer minor units (XAF/EUR/USD branded `Number`, never JS float).
- No `.env`, no `.pem`, no private keys committed; gitleaks config in place.
- Citizen tips have triple-locked retention: DB triggers + append-only history table + repo refusing direct DELETE (DECISION-016).

## Phase 2 — execution rules (recap)

When the architect approves Phase 2:

1. Order: critical → high → medium → low. Within a tier, security-class first.
2. Branch per finding: `audit/AUDIT-NNN-short-slug`.
3. Failing test first; minimum-diff fix; full `pnpm typecheck && pnpm lint && pnpm test` clean before commit.
4. Commit message: `AUDIT-NNN: <one-line summary>`. Body: what was wrong, the original buggy line, which test guards it.
5. Update this `AUDIT.md` per finding: `status: open` → `status: fixed (commit <sha>)`.
6. Final deliverable: `AUDIT-REPORT.md` with original counts, fixed counts, remaining counts, patterns observed, recommended hardening.

One finding / one commit / one test. Do not bundle.
