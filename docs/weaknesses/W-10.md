# W-10 — WebAuthn → secp256k1 path is fragile

| Field | Value |
|---|---|
| ID | W-10 |
| Severity | **Medium** |
| Status | **fix proposed** |
| Authoritative ref (in TRUTH or repo) | TRUTH.md Section E |
| Source(s) in original docs | SRD §17.8.3 |

## Problem

SRD §17.8.3 attempts COSE-alg-`-47` (secp256k1) translation with 'trial recovery' of the v byte. Most browsers do not surface secp256k1 in WebAuthn (Chrome supports it on YubiKey 5+ since firmware 5.4, Firefox/Safari support is variable).

## Fix

**Default path: native desktop helper** (Tauri or Electron-with-strict-CSP) that uses `libykcs11` directly to drive the YubiKey for council vote signing. Reserve WebAuthn for accessibility-edge cases. Council members install the helper at the §13 enrolment ceremony; the architect publishes a reproducible build with EV signature.

WebAuthn fallback remains documented for rare browser-only scenarios but is not the primary path.
