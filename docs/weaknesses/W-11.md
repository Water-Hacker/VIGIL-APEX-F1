# W-11 — Hyperledger Fabric single-peer single-orderer is theatre

| Field | Value |
|---|---|
| ID | W-11 |
| Severity | **Medium** |
| Status | **fix proposed** |
| Authoritative ref (in TRUTH or repo) | TRUTH.md Section B + ROADMAP.md Phase 2 |
| Source(s) in original docs | SRD §3.10, SRD §04.7 |

## Problem

SRD §3.10 + the docker-compose entry for `vigil-fabric` runs Fabric as one peer + raft single-node. A single-peer permissioned ledger provides no Byzantine fault tolerance and no third-party verification — it's a JSON-Lines file with extra steps.

## Fix

**Two options**, recommendation (b):

(a) Add a second Fabric peer running on the backup architect's machine from M0c, joined via WireGuard — gives real BFT signalling.

(b) **(recommended) Drop Fabric from MVP** and replace with a SQL-native append-only hash-chained `audit.actions` table (already half-specified in SRD §7.7). The Polygon anchor remains unchanged either way. Saves ~12 GB RAM, two weeks of provisioning effort, and removes single-peer-equals-no-trust theatre.

Fabric is reintroduced properly at Phase 2 with multi-org (CONAC + Cour des Comptes + VIGIL APEX SAS = 3-peer minimum).
