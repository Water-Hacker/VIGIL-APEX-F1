# VIGIL APEX — FAILURE MODES AUDIT

## Comprehensive Dependency & Component Resilience Analysis

**Prepared for**: Binding Institutional Audit  
**Scope**: All external dependencies and internal components per SRD v3.0

**Methodology**: Code-level verification of failure handling, recovery paths, operator notification, and state consistency.

## EXTERNAL DEPENDENCIES

### 1. PostgreSQL (N03)

| Aspect                                                                                                                                                         | Finding                                                                                                                                                                                                                                           |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Connection Failure**                                                                                                                                         | Loud failure. ioredis pool (`packages/db-postgres/src/client.ts` L95-96) logs all pool errors via `pool.on('error')`. Pool config (L73-94) enforces retry with exponential backoff on reconnect; query_timeout at 30s bounds hanging connections. |
| **Silent vs Loud**                                                                                                                                             | Loud. Failed connections throw immediately; queries against a dead pool fail synchronously.                                                                                                                                                       |
| **Partial Failure**                                                                                                                                            | Supported. Pool maintains min/max connection bounds (default 2/40, tunable). Single connection loss doesn't cascade.                                                                                                                              |
| **Total Failure**                                                                                                                                              | Total. No connection pooling fallback or replica auto-failover in code; operator must manage RDS failover externally. Docker restart policy: `unless-stopped` (compose.yaml L93).                                                                 |
| **Detection**                                                                                                                                                  | Prometheus metrics via `poolStats()` (L112-118); healthcheck (compose.yaml L187-192) runs `pg_isready` every 30s.                                                                                                                                 |
| **Mitigation**                                                                                                                                                 | Statement timeout (30s default, tunable POSTGRES_STATEMENT_TIMEOUT_MS), lock timeout (5s, L101), idle-in-transaction timeout (5min, L102). All per-connection on client side.                                                                     |
| **State After Failure**                                                                                                                                        | Transactions are rolled back on connection loss. Durable writes flushed before pool disconnect.                                                                                                                                                   |
| **Operator Notification**                                                                                                                                      | Sentry integration (if SENTRY_DSN set, .env.example L264) captures pool errors. No on-call alert defined in code; relies on external monitoring.                                                                                                  |
| **Recovery**                                                                                                                                                   | Docker restart waits up to 3 retries × 60s (SRD §3.6 per compose). Application-side: exponential backoff retries in ioredis config (L63-67).                                                                                                      |
| **Gap**: No connection-level circuit breaker; a sustained slow-Postgres condition can exhaust the pool and starve other services. Mitigation is timeouts only. |

### 3. Redis (N05)

| Aspect                                                                                            | Finding                                                                                                                                                                                                         |
| ------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Connection Failure**                                                                            | Loud failure. `packages/queue/src/client.ts` (L88-90) logs every redis error, close, and reconnect event. Retry strategy uses exponential backoff (L63-66: max 5s delay, 3 retries per request).                |
| **Silent vs Loud**                                                                                | Loud. `redis.on('error')` and explicit check in `ping()` (L93-96) throws if response != 'PONG'.                                                                                                                 |
| **Partial Failure**                                                                               | Possible. A slow Redis node (but responsive) causes the connection to stall; reconnect-on-error config (L68-71) re-tries on READONLY or ECONNRESET but NOT on latency > threshold.                              |
| **Total Failure**                                                                                 | Total for queue operations. No producer buffer; if Redis is down, enqueue calls throw immediately. `enableOfflineQueue: false` (L62) ensures fail-fast.                                                         |
| **Detection**                                                                                     | Error events logged with `logger.error()`. Process-level: Prometheus exporter (observability package) exposes queue depth. Container-level: healthcheck every 30s (compose.yaml L259-264).                      |
| **Mitigation**                                                                                    | Exponential backoff, maxRetriesPerRequest=3, reconnectOnError handler for specific error codes. Consumer groups have idle-reclaim (REDIS_CONSUMER_IDLE_RECLAIM_MS=300s, .env L77) to recover from dead workers. |
| **State After Failure**                                                                           | Enqueued messages persist in the stream if at least one consumer has acknowledged them. Unacknowledged messages remain pending-entry-id and are auto-claimed after idle-reclaim window.                         |
| **Operator Notification**                                                                         | Sentry if configured. No in-band alert.                                                                                                                                                                         |
| **Recovery**                                                                                      | Container restart. Streams survive; consumer groups resume from their last acknowledged offset.                                                                                                                 |
| **Gap**: Redis Sentinel/Cluster failover not implemented in Phase 1; no automatic shard failover. |

### 4. Vault (N08)

| Aspect                                                                                                                                                                   | Finding                                                                                                                                                                                                                         |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Connection Failure**                                                                                                                                                   | Loud failure. `packages/security/src/vault.ts` (L39-59) reads the token file at startup and connects to the Vault endpoint. If unreachable, `VaultUnsealError` is thrown immediately at boot.                                   |
| **Token Renewal**                                                                                                                                                        | Auto-renewal implemented (L131-162). Failures logged (L154-161); `vaultTokenRenewFailedTotal` metric incremented on failure. Token TTL lookups may fail silently if the token is already expired, leading to stale credentials. |
| **Silent vs Loud**                                                                                                                                                       | Mostly loud at boot (fail-fast); token renewal failures are logged but workers continue with stale token until the next renewal is attempted. If a token expires between renewals, the next secret read fails loudly (L76).     |
| **Partial Failure**                                                                                                                                                      | Possible. Vault can be read-only; `vault.write()` would fail (L89-93) but `vault.read()` succeeds. Workers requiring secret rotation would fail.                                                                                |
| **Total Failure**                                                                                                                                                        | Total. No fallback secret store. Workers cannot boot without Vault. Durable operations resume from restart, but real-time secret access is blocked.                                                                             |
| **Detection**                                                                                                                                                            | Renewal failures increment Prometheus counter (L160). Sentry on read/write failures (L76-85). No heartbeat/liveness probe in code.                                                                                              |
| **Mitigation**                                                                                                                                                           | Token auto-renewal at 50% TTL (L139: `interval = max(60s, ttl * 500ms)`). Retry on `vault.read()` is left to the caller (typically 1 attempt). No circuit breaker.                                                              |
| **State After Failure**                                                                                                                                                  | Workers holding stale credentials fail on next operation. Restart is required to load fresh token.                                                                                                                              |
| **Operator Notification**                                                                                                                                                | Metric `vaultTokenRenewFailedTotal{service}` surfaces in Prometheus; alert via Alertmanager (configured in compose). No code-level runbook.                                                                                     |
| **Recovery**                                                                                                                                                             | Manual Vault troubleshooting + container restart. No automatic failover to a secondary Vault instance.                                                                                                                          |
| **Gap**: No circuit breaker for Vault; a slow Vault endpoint can stall all secret operations. Renewal failures are logged but not aggregated into a service-level alert. |

## INTERNAL COMPONENTS

### 1. Worker Container Crash (All Workers)

| Aspect                                                                                                                   | Finding                                                                                                                                                                                                                              |
| ------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Restart Policy**                                                                                                       | Loud recovery. `docker-compose.yaml` L92-93: `restart: unless-stopped`. On crash, Docker restarts up to 5 times within 60s, then backs off 30s between attempts. After 3 restart cycles (180s), watchdog raises P1 alert (SRD §3.6). |
| **Detection**                                                                                                            | Docker health probe (compose services define healthcheck every 15-30s). Container exiting with non-zero code is logged.                                                                                                              |
| **State After Failure**                                                                                                  | In-flight async tasks in Redis streams (enqueued but not yet acknowledged) are reclaimed after REDIS_CONSUMER_IDLE_RECLAIM_MS (5min default). Unacknowledged Postgres transactions are rolled back on connection close.              |
| **Operator Notification**                                                                                                | Sentry captures uncaught exceptions. Prometheus counter for container restarts (not shown). Docker events can feed to Alertmanager.                                                                                                  |
| **Recovery**                                                                                                             | Automatic restart with backoff. Manual: tail logs via `docker logs`, verify disk/memory, restart if hung.                                                                                                                            |
| **Gap**: No graceful drain signal (SIGTERM timeout before SIGKILL). Workers consuming long-running tasks may lose state. |

### 3. Single Adapter Failure Isolation

| Aspect                                                                                                                                                                   | Finding                                                                                                                                                                                                                                                                                               |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Isolation**                                                                                                                                                            | Partial. Each adapter runs as a separate invocation in the adapter-runner worker (apps/adapter-runner/src/run-one.ts, not shown in search). If one adapter's run() throws, the runner catches (run-one.ts infers) and logs; next adapter is scheduled. Failure of adapter A does not block adapter B. |
| **Failure Mode**                                                                                                                                                         | Adapter-specific. First-contact failures (HTML fetch fails, CSS selector breaks) mark adapter_health.status='first_contact_failed' and bump consecutive_failures. After 3+ failures, worker-adapter-repair is triggered (worker-adapter-repair/src/index.ts L52-75).                                  |
| **Silent vs Loud**                                                                                                                                                       | Loud. Adapter health is recorded in adapter_health table; operators review the dead-letter queue. Adapter failure does NOT propagate to findings—no rows are inserted.                                                                                                                                |
| **Partial Failure**                                                                                                                                                      | Possible. Rate-limiter cap reached marks adapter as RATE_LIMIT_CAP_REACHED; next run skipped (run-one.ts infers from DailyRateLimiter.allow()).                                                                                                                                                       |
| **Total Failure**                                                                                                                                                        | Total for that adapter. No fallback to secondary source. Adapter runner moves to next source in the registry (infra/sources.json).                                                                                                                                                                    |
| **Detection**                                                                                                                                                            | adapter_health table tracks status, consecutive_failures, last_error, next_scheduled_at. Prometheus `vigil_adapter_runs_total{adapter,outcome}` counter.                                                                                                                                              |
| **Mitigation**                                                                                                                                                           | Rate limiting per source (TIP_ATTACHMENT_LIMITS.daily_request_cap, .env L156). Repair worker (worker-adapter-repair) proposes CSS selector fixes for broken adapters (daily 03:00 cron).                                                                                                              |
| **State After Failure**                                                                                                                                                  | Adapter state is isolated to adapter_health and source.adapter_selector_registry. Cached first-contact HTML persists (/srv/vigil/first-contact/).                                                                                                                                                     |
